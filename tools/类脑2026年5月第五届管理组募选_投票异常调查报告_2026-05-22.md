# 《类脑2026年5月第五届管理组募选》投票异常调查报告

> **密级：内部调查材料 / 不建议公开分发**  
> 本报告基于 SQLite 数据库中的原始投票记录生成。数据库可看到“投票人 ID → 被投候选人 ID”的对应关系，涉及投票隐私。若需对外说明，建议只公开结论摘要和统计结果，不公开投票人 ID 明细。

---

## 0. 结论摘要

针对“有人私下反馈存在小团体拉票”的情况，我检查了最新的 `tools/court522.db`，并与上一份 `tools/court.db` 进行了对比。

综合判断：

- **数据库层面存在强烈异常投票模式。**
- **最突出异常是一个固定 5 人名单被 23 名投票人完全相同地选择。**
- **该 5 人名单还出现在 63 张票中，占全部 258 张投票的 24.4%。**
- 简单随机模拟显示，在保持候选人热度和每人选择人数分布的情况下，类似规模的完全重复票型极难自然产生。
- 这组模式高度符合“推荐名单式投票 / 小团体组织投票 / 拉票名单转发”的特征。
- 但仅凭数据库记录，**不能单独证明存在具体的私下拉票行为**。数据库只能证明投票模式异常，若要定性处罚，仍建议补充聊天记录、私信截图、群聊截图或当事人说明。

本报告中的主要可疑组合为：

```text
𝕯𝖎𝖓𝖌𝖚𝖘 𝕽𝖊𝖝 (理塘の王)
TUOte17
𝓜𝓪𝓻𝓢𝓪𝓷
𝓗𝓪𝓹𝔂𝓶𝓪𝓱𝓮𝓻
Ein
```

以下简称 **G5 组合**。

## 3. 方法说明

### 3.1 使用的数据表

本次调查主要使用以下表：

```text
pe_elections          募选主表
pe_registrations      报名/候选人表
pe_votes              投票器表
pe_vote_records       投票记录表
pe_vote_invalidations 作废投票记录表
pe_fields             岗位字段表
```

核心字段：

```text
pe_vote_records.voter_id           投票人 Discord 用户 ID
pe_vote_records.selected_user_ids  被投候选人 Discord 用户 ID JSON 列表
pe_vote_records.created_at         投票提交时间，UTC
```

### 3.2 时间说明

数据库中的 `created_at` 是 UTC 时间。报告中除特别说明外，均换算为：

```text
UTC+8，北京 / 香港 / 澳门 / 台湾时间
```

### 3.3 分析方法

本次检查了：

1. 投票数量变化；
2. 候选人得票统计；
3. 完全相同票型；
4. 固定组合共同出现次数；
5. 时间集中度；
6. 与旧数据库的新增投票对比；
7. 简单随机模拟；
8. 移除可疑票型后的结果影响。

### 3.4 重要限制

数据库可以证明：

- 谁投了票；
- 投给了谁；
- 何时投票；
- 是否出现大量相同票型。

数据库不能单独证明：

- 是否有人私下发起拉票；
- 是否存在群聊组织；
- 候选人本人是否知情；
- 投票人是否受到诱导、承诺或胁迫。

因此，本报告给出的结论是“异常模式强烈支持存在组织化投票可能”，而不是“已经证明某人违规拉票”。

## 5. 主要异常发现

## 5.1 异常一：G5 组合出现 23 张完全相同票型

出现次数最高的完全相同票型为：

```text
𝕯𝖎𝖓𝖌𝖚𝖘 𝕽𝖊𝖝 (理塘の王)
TUOte17
𝓜𝓪𝓻𝓢𝓪𝓷
𝓗𝓪𝓹𝔂𝓶𝓪𝓱𝓮𝓻
Ein
```

该票型的特征：

| 项目 | 值 |
|---|---:|
| 完全相同票数 | 23 |
| 占全部 258 张投票比例 | 8.9% |
| 每张票选择人数 | 5 |
| 是否包含其他候选人 | 否 |
| 首次出现 | 2026-05-20 09:08:47 |
| 最新出现 | 2026-05-21 23:50:28 |

这个异常的核心不只是“五人都很热门”，而是：

> 有 23 个投票人提交了完全相同的 5 人名单，且没有添加任何其他候选人。

在多候选、多选项投票中，大量投票人选择完全相同的 5 人名单，通常更接近“照着同一份推荐名单投票”，而不是自然独立偏好。

---

## 5.2 异常二：G5 组合被 63 张票完整包含

除了 23 张完全相同的 G5 票外，还有更多投票虽然额外选择了其他候选人，但仍完整包含了 G5 组合。

| 项目 | 值 |
|---|---:|
| 包含完整 G5 的票数 | 63 |
| 占全部 258 张投票比例 | 24.4% |

也就是说，大约每 4 张票中就有 1 张完整包含这 5 人。

这说明 G5 不只是少量完全重复票型，而是形成了一个显著的共同投票核心。

## 7. 简单随机模拟结果

为了判断 G5 异常是否可能自然产生，我做了一个简单随机模拟。

### 7.1 模拟方法

每次模拟保持：

- 总投票人数：258；
- 每张票的选择人数分布不变；
- 候选人总体热度作为抽样权重；
- 每张票内不重复选择候选人。

模拟次数：

```text
2000 次
```

该模型并不完美，但它已经尽量照顾了“热门候选人更容易一起出现”的情况。

### 7.2 模拟结果

| 指标 | 实际观察值 | 模拟均值 | 模拟 P95 | 模拟 P99 | 模拟最大值 | 达到或超过实际值次数 |
|---|---:|---:|---:|---:|---:|---:|
| 最大完全重复票型数 | 23 | 6.371 | 9 | 10 | 14 | 0 / 2000 |
| G5 完全相同票数 | 23 | 0.025 | 0 | 1 | 1 | 0 / 2000 |
| 包含完整 G5 的票数 | 63 | 15.387 | 21 | 23 | 28 | 0 / 2000 |

### 7.3 模拟结论

即使允许热门候选人更容易被抽到，模拟中也没有出现接近实际 G5 异常规模的情况。

这说明：

> G5 的出现频率和完全重复程度，难以用普通随机偏好解释。

## 10. 调查判断

### 10.1 证据强度评估

| 证据项 | 强度 | 说明 |
|---|---|---|
| 23 张完全相同 G5 票 | 高 | 多候选多选投票中，固定 5 人名单被大量完全复刻，异常显著 |
| 63 张票完整包含 G5 | 中高 | 显示 G5 是共同投票核心，不只是孤立重复 |
| 时间集中段 | 中 | 与推荐名单转发后集中投票相符，但不能单独定性 |
| 随机模拟结果 | 中高 | 2000 次模拟无一接近实际异常规模 |
| 新增票中又出现 1 张 G5 完全票 | 中 | 说明该模式在最新数据中仍延续 |
| 候选人之间互投 | 低到中 | 多选制中候选人互投常见，单独证据力有限 |

### 10.2 综合结论

本次投票中，G5 组合存在显著异常：

```text
固定五人名单被 23 名投票人完全相同选择；
该五人组合被 63 张票完整包含；
其出现频率远高于简单随机模型可解释范围；
且对当前排名有实质影响。
```

因此可以认为：

> 存在较强的“小团体推荐名单式投票 / 组织化投票”可能。

但是否构成“违规拉票”，还需要结合募选规则和站内外证据判断。

### 11.2 向举报人补充取证

重点索取：

- 群聊推荐名单截图；
- 是否有人明确发出上述 G5 五人名单；

  - 2026-05-20 09:08 - 09:49；
  - 2026-05-20 12:35 - 13:01；

### 11.3 询问相关候选人或管理员

建议先内部询问，不建议一开始公开点名所有投票人。

可询问：

- 是否有人制作或传播推荐名单；
- 是否有组织成员统一投票；
- 候选人是否参与拉票；
- 是否有承诺、交换、威胁、利益诱导。

### 11.4 处理建议

如果规则中**明确禁止拉票 / 组织投票 / 小团体操纵**：

- 可以将 G5 异常作为启动调查的主要依据；
- 结合聊天证据后决定是否：
  - 警告；
  - 公示异常并延长调查期；
  - 作废特定违规票；
  - 重启投票；
  - 取消相关候选人资格。

如果规则中**没有明确禁止拉票**：

- 不建议仅凭统计异常直接处罚；
- 可以发布透明说明，指出存在异常集中票型；
- 后续补充规则，例如禁止私下组织统一名单、禁止候选人私聊拉票、要求推荐公开透明等。

```sql
SELECT
  id,

```

### 14.2 查看投票记录数量和最新投票时间

```sql
SELECT
  COUNT(*) AS vote_records,
  MAX(id) AS max_record_id,
  MIN(created_at) AS first_vote_time,
  MAX(created_at) AS latest_vote_time
FROM pe_vote_records
WHERE election_id = 6;
```

### 14.3 展开“谁投给谁”

```sql
SELECT
  vr.id AS record_id,
  vr.voter_id,
  vr.created_at,
  j.value AS selected_user_id,
  c.display_name AS selected_display_name
FROM pe_vote_records vr
JOIN json_each(vr.selected_user_ids) j
LEFT JOIN pe_registrations c
  ON c.election_id = vr.election_id
 AND c.user_id = CAST(j.value AS INTEGER)
WHERE vr.election_id = 6
ORDER BY vr.created_at ASC, vr.id ASC;
```

### 14.4 查看候选人当前得票

```sql
SELECT
  c.user_id AS candidate_user_id,
  c.display_name AS candidate_name,
  COUNT(*) AS votes
FROM pe_vote_records vr
JOIN json_each(vr.selected_user_ids) j
LEFT JOIN pe_registrations c
  ON c.election_id = vr.election_id
 AND c.user_id = CAST(j.value AS INTEGER)
WHERE vr.election_id = 6
GROUP BY c.user_id, c.display_name
ORDER BY votes DESC;
```

## 15. 最终意见

基于 `tools/court522.db` 的投票记录，本次募选存在较强的异常集中投票现象，尤其是 G5 组合的 23 张完全相同票和 63 张完整包含票。该模式已经超出普通热门候选人共同被选的正常解释范围。

建议：

1. 先将本报告和数据库副本作为内部证据保全；
2. 向举报人索取聊天/私信/群聊截图；
3. 对 G5 相关候选人及可能组织者进行内部询问；
4. 在没有直接拉票证据前，不建议仅凭数据库统计直接公开处罚；
5. 若补充证据显示确有组织拉票，再根据既有规则决定是否作废特定票、重启投票或处理相关候选人。

数据库证据显示“有小团体或推荐名单式投票”的可能性很高；但要定性“违规拉票”，仍需要外部沟通证据补强。
